# Requirements Document: Jenkins Log Failure Analyser

## 1. Purpose

Develop a command-line tool that reads a Jenkins build log file, analyses its
content to determine **why the build failed**, and produces a structured
diagnostic report with a root-cause summary, relevant log excerpts, and
actionable remediation hints.

## 2. Scope

- A single Python module `jenkins_analyser.py` plus a `detectors.py` module
  containing the individual failure-detection rules.
- Input is a plain-text Jenkins log file (the raw console output downloaded
  from Jenkins or saved locally).
- The tool works entirely offline — no Jenkins API calls.
- Output is a Markdown report (`.md`) written to a file or printed to stdout.

## 3. Functional Requirements

### 3.1 Command-Line Interface

```
python jenkins_analyser.py --log <build.log> [--output <report.md>] [--verbose]
```

| Flag | Required | Default | Description |
|------|----------|---------|-------------|
| `--log` | yes | — | Path to the Jenkins console log file |
| `--output` | no | stdout | Path for the Markdown report; omit to print to stdout |
| `--verbose` | no | off | Include full matching log excerpts (not just snippets) |

Exit codes: `0` success (report generated), `1` runtime error, `2` usage error.

### 3.2 Failure Detection

The tool must detect and classify the following failure categories.
Multiple categories may fire for a single log.

| ID | Category | Detection Signal |
|----|----------|-----------------|
| `BUILD_TOOL` | Build tool failure | `BUILD FAILURE`, `BUILD FAILED`, `make: ***`, `ninja: build stopped` |
| `TEST_FAILURE` | Unit/integration test failure | `Tests run:.*Failures:` (Maven), `FAILED` (pytest), `AssertionError`, `FAIL:` (JUnit XML summary) |
| `COMPILE_ERROR` | Compilation / syntax error | `error:` lines from `javac`/`gcc`/`clang`/`tsc`, `SyntaxError:`, `cannot find symbol` |
| `DEPENDENCY` | Missing or unresolvable dependency | `Could not resolve`, `Package not found`, `ModuleNotFoundError`, `npm ERR!`, `error: failed to download` |
| `OOM` | Out-of-memory / resource limit | `java.lang.OutOfMemoryError`, `Killed` (OOM killer), `MemoryError` |
| `TIMEOUT` | Step or pipeline timeout | `Timeout waiting`, `timed out after`, `SIGTERM` following a timeout message |
| `INFRA` | Infrastructure / environment issue | `Connection refused`, `No such file or directory` (on required tools), `command not found`, Docker/K8s pull errors |
| `CONFIG` | Misconfiguration | `Invalid configuration`, `Unknown property`, `YAML parse error`, `is not a valid value for` |
| `UNKNOWN` | No pattern matched | Fallback when no detector fires |

Detection is performed in order; all matching categories are reported (not
just the first).

### 3.3 Evidence Extraction

For each matched category the tool must extract:

- **Trigger line** — the exact line that matched the detection pattern.
- **Context window** — up to 10 lines before and after the trigger line
  (configurable via `--context N`, default 10).
- **Line number** — 1-based line number within the log file.

### 3.4 Report Structure

The Markdown report must contain:

```
# Jenkins Build Failure Report

**Log file:** <path>
**Analysed at:** <ISO-8601 timestamp>
**Build result:** FAILURE  (or SUCCESS if no failure detected)

## Summary

<One-paragraph plain-English explanation of what went wrong, synthesised
from all matched categories. If multiple categories fired, describe the
likely causal chain (e.g. "A missing dependency caused the compile step
to fail, which in turn caused the build tool to report failure.")>

## Detected Failure Categories

### 1. <Category Name> (`<ID>`)

**Trigger line <N>:**
```
<trigger line>
```

**Context:**
```
<context window>
```

**Remediation hint:** <one or two sentences suggesting how to fix this class of failure>

### 2. …

## Remediation Checklist

- [ ] <Actionable item derived from each detected category>

---
*Report generated by jenkins_analyser.py*
```

### 3.5 Remediation Hints

Each failure category ships a default remediation hint:

| ID | Hint |
|----|------|
| `BUILD_TOOL` | Check the build tool output above the failure line for the root error; ensure all required targets/goals are defined. |
| `TEST_FAILURE` | Review the failing test names in the log; run them locally with `--verbose` to reproduce. |
| `COMPILE_ERROR` | Fix the reported compile errors; check that the correct SDK/compiler version is installed. |
| `DEPENDENCY` | Verify the dependency name and version; check network access to the package registry. |
| `OOM` | Increase the JVM heap (`-Xmx`) or runner memory limits; consider splitting large test suites. |
| `TIMEOUT` | Increase the step timeout in the Jenkinsfile or investigate why the step is slow. |
| `INFRA` | Check that required tools are installed on the agent and that external services are reachable. |
| `CONFIG` | Validate the configuration file against its schema; check for typos or removed keys. |
| `UNKNOWN` | Inspect the last 50 lines of the log manually for the failure signal. |

### 3.6 Error Handling

| Error condition | Expected behaviour |
|---|---|
| Log file not found | Print error to stderr; exit 1 |
| Log file is empty | Print warning; emit report with `BUILD_RESULT: UNKNOWN`; exit 0 |
| Output path not writable | Print error to stderr; exit 1 |
| Wrong CLI arguments | Print usage to stderr; exit 2 |

## 4. Non-Functional Requirements

- **No external dependencies** — standard library only (`re`, `argparse`,
  `pathlib`, `datetime`, `textwrap`).
- **Modular** — each detector is an independent function in `detectors.py`
  returning a list of `Match` objects; `jenkins_analyser.py` orchestrates and
  renders.
- **Extensible** — adding a new failure category requires only adding a new
  detector function and registering it in the detector list; no changes to the
  orchestrator.
- **Tested** — `tests/test_jenkins_analyser.py` includes:
  - One test per detection category using a minimal synthetic log snippet.
  - An end-to-end test on a realistic multi-failure sample log.
  - A test confirming that a clean (passing) log produces `BUILD_RESULT: SUCCESS`.

## 5. Sample Inputs

The `tests/fixtures/` directory must contain at minimum:

| File | Description |
|------|-------------|
| `maven_test_failure.log` | Maven build with failing JUnit tests |
| `npm_missing_dep.log` | Node build with an unresolvable npm dependency |
| `oom_java.log` | Java build killed by OOM |
| `clean_build.log` | A fully successful build log |

These fixtures are used directly by the test suite.

## 6. Acceptance Criteria

1. Running on `maven_test_failure.log` detects at least `TEST_FAILURE` and
   `BUILD_TOOL` categories.
2. Running on `npm_missing_dep.log` detects `DEPENDENCY` and `BUILD_TOOL`.
3. Running on `oom_java.log` detects `OOM`.
4. Running on `clean_build.log` produces a report with `BUILD_RESULT: SUCCESS`
   and no failure categories.
5. The report for `maven_test_failure.log` contains a non-empty Summary
   paragraph and a populated Remediation Checklist.
6. `--output report.md` writes to file; omitting it prints to stdout.
7. All unit tests pass (`python -m pytest tests/test_jenkins_analyser.py`).
